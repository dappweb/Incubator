# DApp 测试网上线路线图（按当前仓库修订）

## 1. 目标定义

当前仓库状态不是“生产就绪”，而是“链上 MVP 骨架”阶段：
- 已有基础合约：矿机购买、节点/超级节点购买、OTC 基础挂单/成交
- 已有基础前端：连接钱包、Sepolia 网络校验、矿机购买入口、公告读取
- 未完成核心业务：分账、奖励、身份 NFT 闭环、订单总账、OTC 市场页、自动化测试

**本阶段目标不是主网上线，而是达到 Sepolia 测试网可验收状态。**

测试网上线定义：
1. 业务主流程能端到端跑通。
2. 业务口径与需求文档一致，不再出现“产品说法”和“链上实现”不一致。
3. 关键风险点可控，至少具备暂停、参数边界、事件可追溯、测试回归四项能力。

---

## 2. 业务基线（本次必须对齐的范围）

依据 [docs/DAPP_UI_SPEC.md](docs/DAPP_UI_SPEC.md) 与现有业务文档，测试网 V1 必须对齐以下范围：

### P0：测试网上线必做
- 矿机购买：1-10 台、USDT 支付、订单落链、分账事件可追溯
- 节点购买：限购 1 个、身份状态可读取
- 超级节点购买：限购 1 个、身份状态可读取
- OTC：身份资产可挂单、撤单、成交，收取 10% 手续费
- 公告：Appwrite 只承载公告，不承载订单/奖励/身份账本
- 前端：网络校验、授权提示、交易状态反馈、基础明细展示

### P1：测试网建议同步完成
- 矿机分账比例：60/5/5/8/20/2
- 关键标准事件：`MachinePurchased`、`NodePurchased`、`SuperNodePurchased`、`PoolAllocated`、`OtcOrderCreated`、`OtcOrderFilled`、`OtcOrderCancelled`
- 基础 view 查询：价格、角色、订单列表、订单详情、OTC 挂单列表、上次成交价

### P2：可以延后到主网前
- 多签 + Timelock
- 奖励复杂结算（如直推循环、团队加权、烧伤）
- 审计与监控平台接入
- 前端首页收益看板、团队收益排行、收益试算器

## 3. 当前代码与业务需求差距

| 模块 | 当前仓库现状 | 与业务要求的差距 | 是否阻塞测试网上线 |
|---|---|---|---|
| `IncubatorCore.sol` | 仅完成基础购买与角色记录 | 没有分账、没有池子归集、没有订单详情查询、没有暂停机制、没有参数上限 | 是 |
| `NodeOTCMarket.sol` | 有挂单/撤单/成交/费率 | 没有身份 NFT 集成约束、没有挂单枚举、没有市场查询 view、没有防重复挂单设计 | 是 |
| 奖励模块 | 未实现 | 没有奖励发放、没有奖励总账事件 | 对当前“最小测试网”非阻塞，对业务闭环阻塞 |
| 前端 `App.tsx` | 只有钱包、矿机购买、公告 | 缺少授权流、角色展示、节点/超级节点页、OTC 页、订单明细、风险确认 | 是 |
| 数据可观测 | 只有零散事件 | 无法独立核对分账与订单结果 | 是 |
| 测试 | 当前无自动化测试 | 无法作为测试网上线门槛 | 是 |
| 部署 | 有基础脚本 | 缺少 Mock USDT、验收脚本、环境模板、部署记录 | 是 |

结论：**当前仓库距离“上测试网可验收”仍差一个 P0 补齐阶段，不建议直接部署给外部验证。**

## 4. 测试网上线必须补齐的能力清单

## 4.1 合约侧（P0）

### Core
- 增加矿机分账落链：
	- 60% LP/底池
	- 5% 直推池
	- 5% 超级节点池
	- 8% 节点池
	- 20% 平台池
	- 2% 榜单池
- 增加池子地址/池子余额配置与更新接口
- 所有比例加总校验必须等于 100%
- 所有参数增加上限校验，防止误配
- 增加 `pause/unpause`
- 增加 `getMachineOrder(uint256)`、`getUserMachineOrders(address)`、`getUserRole(address)` 等只读接口

### Node / SuperNode
- 节点限购 1 个
- 超级节点限购 1 个
- 明确升级关系：
	- 是否允许 `None -> SuperNode`
	- 是否要求 `Node -> SuperNode`
- 该规则必须写死到 PRD 与合约一致

### OTC
- 明确交易标的是“身份 NFT”还是“链上映射身份凭证”
- 若继续走 NFT 方案，测试网前必须补：
	- 身份 NFT 合约
	- 上架前 `approve`
	- 下架/成交后的所有权一致性
- 增加市场读取接口：
	- `getOrder(orderId)`
	- `getActiveOrderIds()` 或分页查询
	- `getLastTradePrice(nftContract)`
- 防止同一资产重复挂单

### 事件标准
- 所有关键写操作必须带上：`user`、`orderId/tokenId`、`amount`、`token`、`blockNumber` 可推导字段
- 分账必须发出独立事件，不能只留购买事件

## 4.2 前端侧（P0）

- 购买矿机流程拆成两步：
	1. USDT 授权
	2. 购买提交
- 所有写交易增加风险确认弹窗，至少展示：
	- 数量/价格
	- 手续费/分账预览
	- 当前网络
	- 钱包地址
- 新增节点页：价格、状态、购买按钮、限购说明
- 新增超级节点页：价格、状态、升级按钮、身份说明
- 新增 OTC 页：挂单列表、创建挂单、撤单、买入
- 新增我的资产区：
	- 我的角色
	- 我的矿机订单
	- 我的 OTC 订单
	- 最近交易状态
- Appwrite 故障时，除公告外其余功能不得受影响

## 4.3 工程与测试（P0）

- 增加 Mock USDT 合约，用于本地与 Sepolia 联调
- 增加合约测试：
	- 矿机购买成功/失败
	- 分账比例正确
	- 节点/超级节点限购
	- OTC 挂单/撤单/成交
	- 价格下限约束
- 增加最小回归脚本：
	- 部署
	- 初始化参数
	- 铸造测试资产
	- 跑一轮端到端验收
- 输出 `.env.example` 与测试网部署记录模板

## 5. 推荐排期（以“2-3 周上 Sepolia”为目标）

## 第 1 周：业务口径冻结 + 合约补齐

### 产品/业务
- 冻结三件事：
	- 节点升级路径
	- OTC 交易标的形态（NFT / 身份凭证）
	- 测试网阶段是否先不上复杂奖励

### 合约
- 完成 Core 分账
- 增加参数边界与暂停能力
- 增加订单与角色读取接口
- 明确 Node / SuperNode 购买状态机

**周验收**
- 一笔矿机订单可以完整还原分账结果
- 节点/超级节点购买规则不可歧义

---

## 第 2 周：OTC 闭环 + 前端闭环

### 合约
- 完成 OTC 查询接口
- 完成身份资产交易约束
- 补足关键事件

### 前端
- 完成授权 + 购买双步骤
- 完成节点/超级节点页
- 完成 OTC 页最小闭环
- 完成“我的订单/我的角色”

**周验收**
- 用户可以完成：连接钱包 -> 授权 -> 购买矿机 -> 购买节点 -> OTC 上架/买入

---

## 第 3 周：测试、部署、验收

### QA
- 跑通单元测试与场景测试
- 形成测试网验收报告

### 运维
- Sepolia 部署
- 初始化参数
- 配置前端环境变量
- 记录部署地址与版本号

### 验收演示
- 演示 4 条主流程：
	1. 矿机购买
	2. 节点购买
	3. 超级节点购买
	4. OTC 成交

**周验收**
- 外部测试人员可按文档独立复现流程
- 所有关键结果可通过事件或 view 独立验证

## 6. 测试网上线 Gate

## Gate A：功能 Gate
- 矿机、节点、超级节点、OTC 四条主流程可跑通
- 前端不再只有单页购买，而是具备最小业务闭环

## Gate B：一致性 Gate
- 合约分账口径与产品文档一致
- 前端展示口径全部来自链上 view / event
- Appwrite 中不存在业务账本写入路径

## Gate C：质量 Gate
- 合约关键用例测试通过
- 至少一轮端到端联调通过
- 所有失败交易都有明确错误提示

## Gate D：发布 Gate
- 已完成 Sepolia 地址登记
- 已输出环境变量清单
- 已输出回滚方案（前端回滚 + 合约暂停）

## 7. 建议直接派工的任务拆分

### 合约组
1. 先补 `IncubatorCore` 分账、池子、暂停、读取接口。
2. 明确节点/超级节点身份状态机。
3. 补 `NodeOTCMarket` 的查询接口与身份资产约束。

### 前端组
1. 先完成授权流与矿机订单明细。
2. 再补节点/超级节点页面。
3. 最后补 OTC 列表与我的资产页。

### QA 组
1. 建 Mock USDT。
2. 围绕“授权 -> 下单 -> 分账 -> 身份变更 -> OTC 成交”写回归。
3. 输出测试网验收单。

### 运维/产品
1. 固化 Sepolia 参数表。
2. 准备公告内容与测试网引导文档。
3. 输出演示账号、验收场景、异常处理 SOP。

## 9. 一句话结论

如果目标是“尽快上测试网”，建议**先把复杂奖励和主网级安全治理降为 P2**，集中补齐：
- Core 分账
- 身份状态机
- OTC 资产闭环
- 前端最小业务闭环
- 自动化测试与部署验收

做到这 5 项，仓库就能从“演示版 MVP”进入“可对外联调的 Sepolia 测试版”。
